# HardKAS v0.7.1-alpha — Final Engineering Release Report

## 1. Deterministic Runtime Maturity

The HardKAS local BlockDAG execution engine has reached mathematical maturity. By decoupling mock simulation structures from external network calls, we have achieved a fully air-gapped simulated execution environment.

### Transaction Planning Canonicalization Invariants:
1. **Pre-Selection Canonical Sort**: All candidate UTXOs are sorted by value (`amountSompi ASC`) and outpoint (`txId ASC`, `index ASC`) before coin selection, resolving planning divergence caused by unstable RPC UTXO payload orderings.
2. **Post-Selection Canonical Sort**: The final selected UTXO set is re-sorted to neutralize insertion ordering differences introduced by greedy coin-selection loops.
3. **Recipient Output Sorting**: Recipient outputs are sorted (`amountSompi ASC`, `address ASC`) to eliminate volatile output indexing.
4. **Isolated Change Positioning**: The generated change output is strictly appended last (`index = outputs.length`), providing predictable index referencing.

All volatile fields (`rpcHost`, `latencyMs`) are successfully filtered out by the canonical serializer (`canonical.ts`), guaranteeing 100% stable, cross-platform plan hashes (`1cd118fdefc3afefdd176f96ef6a6de85d58dabede91bff0189d4dfc6bdb6bf4`).

## 2. Replay Reliability Status

The timeline state time-travel mechanism (`reconstructStateAtDaa`) has been fully audited. 
* **State Reconstruction**: The local BlockDAG state can safely traverse backward and forward through any historical DaaScore by strictly inverting UTXO transactions (nullifying creations and resuscitating consumptions).
* **Lineage Verification**: Replay verification (`sdk.replay.verify`) performs exhaustive causal validation, ensuring the cryptographic parent-child chain (`preStateHash` -> `postStateHash`) is completely unbroken and free from state contamination.

## 3. Security Hardening Summary

The local dev-server (port `7420`) has transitioned from a highly vulnerable local interface into a hardened, sandboxed daemon.

### Defenses Implemented:
* **Host Header Verification**: Complete protection against DNS Rebinding attacks by strictly whitelisting local host headers (`localhost:7420`, `127.0.0.1:7420`, `[::1]:7420`).
* **CSRF Mutation Headers**: All mutating REST methods (`POST`, `PUT`, `DELETE`) require a custom header check (`X-Hardkas-Request: true`), protecting local workspaces from malicious cross-tab scripts.
* **Cryptographic Authorization Tokens**: Access to `/api/*` endpoints and Server-Sent Event (SSE) streams requires validation of a high-entropy random bearer session token. The dev-server automatically injects this token into the dashboard client during initial HTML bootstrapping (`index.html`), maintaining 100% developer transparency with zero manual authorization friction.

## 4. Dashboard Maturity

The dashboard React application (`apps/dashboard`) is structurally integrated with the local node:
* **Projection Awareness**: React SWR cache is synchronized with disk mutations via SSE messages (`projection-stale`, `projection-synced`) and HTTP headers (`X-Hardkas-Generation`), completely eliminating race conditions.
* **User Feedback**: An elegant, non-blocking notification banner informs the user of brief reindex debounces instead of freezing the UI.
* **Visual Stability**: Visually stabilized and audited under Playwright across all key routes.

## 5. Workflow Engine Maturity

The Sandboxed Workflow Runtime is fully operational:
* **Agent-Mode Sandboxing**: Agential integrations are shielded by hard options (`mode: "agent"`). A strict validation checker (`enforcePolicy`) triggers ppanic exceptions (`POLICY_VIOLATION`) if an automated agent attempts unpermitted network calls, mainnet mutations, or live broadcasts without dry-runs.
* **Linaged Orchestration**: Every multi-step workflow generates a strict, sealed JSON artifact (`hardkas.workflow.v1`) that documents the executed sequence, policy proofs, and cryptographic IDs of all child artifacts produced.

## 7. Operational Limitations & Remaining Technical Debt

* **Bridge Event Reorg Debounce**: Synchronizing Igra L2 bridge events during deep virtual DAG reorgs can cause temporary thread locking.
* **Trace Commands**: `hardkas tx trace` remains suspended. Rebuilding trace generation mapping is planned for the subsequent development milestone.
* **Database Contraction**: Massive automated transaction testing requires periodic SQLite query-store rebuilds (`hardkas query store rebuild`) to compact the WAL logs.
