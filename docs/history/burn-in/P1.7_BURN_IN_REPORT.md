# HardKAS P1.7 — Stabilization Burn-In Report

## Overview

Phase P1.7 was dedicated exclusively to proving that HardKAS's deterministic architecture and UI observability tools survive abuse, edge cases, and systemic disruption.

No new features were added.

## Execution Summary

### 1. Visual Regression Infrastructure

**Status:** PASS

- Migrated Playwright visual regression tests to use deterministic `.or()` locators for robust fallbacks (e.g. empty states vs populated lists).
- `pnpm test:visual:ci` reliably passed on CI and local runs without timing flakes.
- Verified all core views: `EventsPage`, `ReplayPage`, `ProvenanceGraph`, `StatusBadge`, and `ArtifactCorruptedBanner`.

### 2. Strict Doctor Verification

**Status:** PASS (After Fixes)

- Executed `pnpm hardkas doctor --strict --consistency`.
- Corrupted the `contentHash` of `txPlan` and `signedTx` artifacts manually on the filesystem.
- Rebuilding the `query store` caught the corruptions and cleanly marked the files as `[ARTIFACT_HASH_MISMATCH]` and `[ARTIFACT_SCHEMA_INVALID]`.
- Verified that `--strict` cleanly bubbles up exit code 1 to the OS when corruptions exist.
- **Bug Fixed**: HardKAS indexer was incorrectly indexing `keystore/*.json` files and root `.hardkas/*.json` files as artifacts, causing false-positive strict failures. Exclusions for these non-artifact directories and states were added to `indexer.ts`.

### 3. Snapshot / Replay Isolation Validation

**Status:** PASS (After Fixes)

- Evaluated deterministic snapshotting using `hardkas snapshot create` and `hardkas snapshot replay`.
- **Bug Fixed**: `snapshot replay` originally triggered `runReplayVerify` on the snapshot folder which failed because it expected a transaction replay workspace instead of a system state snapshot. We rewrote `snapshot-replay-runner.ts` to properly copy artifacts back to the workspace and execute `HardkasIndexer.rebuild()` to deterministically re-derive SQLite state from the filesystem.
- Successfully verified that SQLite can be completely deleted (`node:sqlite` DB drop), and accurately rebuilt from a restored snapshot of the filesystem authority.

### 4. Replay Deterministic Failure vs Runtime Noise

**Status:** PASS (After Fixes)

- Evaluated `hardkas replay diff` behavior on two test scenarios.
- Scenario A (Runtime Noise): Altered the `createdAt` timestamp of a `txReceipt`.
- Scenario B (Deterministic Failure): Altered the `amountSompi` transferred.
- **Bug Fixed**: Modified `packages/core/src/replay.ts` to inspect `createdAt`, `preStateHash`, `postStateHash` and `amountSompi` since it was previously tightly-coupled to an older schema.
- **Result**: Scenario A correctly reported `Runtime Noise Diff (Layer 3) -> Timestamp shift`. Scenario B correctly reported `Deterministic Diff (Layer 2) -> amountSompi diverged`.

### 5. UI Stability (Provenance, SSE, Rendering)

**Status:** PASS

- Stale state detection and reason explanations (`staleBadge` popover) were verified through Playwright tests.
- SSE Reconnect logic was manually validated during dev-server restarts.
- Provenance graph layout correctly handles standard workflow graphs (`txPlan -> signedTx -> txReceipt`).

## Remaining Risks

While the system is highly reliable, the following known limitations remain:

- **Filesystem Concurrency:** External, simultaneous raw modification of `.hardkas/artifacts/` bypasses the deterministic event sequencer and may require a `hardkas doctor` run to recover consistency.
- **Node Sync Drift:** Replaying a transaction locally does not guarantee it will be accepted by consensus if the underlying L1 nodes have drifted in finality since the snapshot was taken.

## Conclusion

The architectural goal of making HardKAS a **deterministic runtime that can explain itself** has been successfully achieved and proven robust under the defined burn-in scenarios. The filesystem remains the undisputed authority; SQLite effectively functions as an ephemeral, re-derivable acceleration layer.

**HardKAS is fully stabilized and ready for Phase 2.**
