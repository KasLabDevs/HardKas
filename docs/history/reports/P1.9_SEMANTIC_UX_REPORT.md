# HardKAS P1.9: Semantic UX & Terminology Audit Report

## Objective
Review core terminology within the HardKAS developer ecosystem for ambiguity, cognitive overload, discoverability, and onboarding clarity, as requested during the P1.9 UX Hardening phase.

---

## 1. Terminology Analysis

### `artifact`
- **Current Meaning**: The absolute, immutable source of truth in HardKAS (tx-plans, receipts, signed-txs).
- **UX Verdict**: **Excellent**. It accurately conveys that this is a persistent, static object that should not be edited by hand but generated by a tool. It feels native to developers (e.g., CI/CD artifacts).

### `projection`
- **Current Meaning**: Derived state indexed from artifacts (currently SQLite query-store).
- **UX Verdict**: **Overloaded & Abstract**. While technically precise in CQRS/Event-Sourcing architectures, "projection" implies complex stream processing to a normal developer.
- **Recommendation**: Transition UI and onboarding language to **"Indexed State"** or **"Query Cache"**. Developers inherently understand that an index or cache can be wiped and rebuilt from the source of truth, perfectly mirroring our artifact/SQLite relationship.

### `replay`
- **Current Meaning**: Deterministic re-execution of the events that led to an artifact.
- **UX Verdict**: **Strong but dangerous**. Developers understand replay, but they often assume it mutates external state (e.g., replaying a failed HTTP request).
- **Recommendation**: Always prefix with **"Deterministic Replay"** in the CLI. The recent `hardkas explain` and `tx send` UX updates correctly frame it as a local deterministic validation scope, removing the danger.

### `snapshot`
- **Current Meaning**: A portable zip/folder containing artifacts and SQLite db for local fast-booting.
- **UX Verdict**: **Ambiguous**. In crypto, a "snapshot" usually implies a global network state capture or consensus proof. 
- **Recommendation**: The recent CLI update explicitly states `Notice: Snapshots are portable local deterministic captures, NOT consensus proofs`. This resolves the ambiguity without changing the term.

### `stale`
- **Current Meaning**: The causal chain of an artifact is broken (e.g., parent hash mismatch), rendering the artifact corrupted in the deterministic graph.
- **UX Verdict**: **Highly Misleading**. "Stale" in web dev means a cache expired (e.g., React Query). In HardKAS, "stale" means cryptographic invariant violation.
- **Recommendation**: Transition the term to **"Orphaned"** (if parent is missing) or **"Invalidated"** / **"Causally Broken"** (if parent hash changed). "Stale" minimizes the severity of the cryptographic failure.

### `simulated` (formerly `simnet`)
- **Current Meaning**: Pure local execution without a Kaspa node, RPC, or Docker.
- **UX Verdict**: **Massive Improvement**. "Simnet" sounded like an actual network. "Simulated" correctly implies it is a local mock execution.
- **Recommendation**: Keep it. It perfectly aligns with the deterministic replay engine.

### `localnet`
- **Current Meaning**: A local instance of state that can involve actual local nodes or simulated execution.
- **UX Verdict**: **Confusing**. It overlaps heavily with `simulated`.
- **Recommendation**: Deprecate the standalone concept of `localnet` in favor of just specifying the `network` and `mode`. If a developer is running a local node, it's `kaspa-node` mode. If they aren't, it's `simulated` mode.

### `provenance`
- **Current Meaning**: The cryptographic lineage and history of how an artifact came to be.
- **UX Verdict**: **Excellent**. It feels professional, academic, and implies high security. It works beautifully in the Dashboard UI.

### `deterministic`
- **Current Meaning**: Re-running the same inputs always yields the same outputs (and hashes).
- **UX Verdict**: **Necessary but Heavy**. 
- **Recommendation**: Keep using it, but pair it with causality blocks in the CLI (which we did in P1.9). Determinism is a promise; the causality block is the proof.

### `correlationId`
- **Current Meaning**: The ID binding an intent (plan), execution (signed tx), and finality (receipt) together in the event timeline.
- **UX Verdict**: **Good**, but purely infrastructural.
- **Recommendation**: Hide it from the casual CLI user. Expose it in `hardkas explain` as the "Lineage Root" or "Execution Trace ID".

## 2. P1.9 Implementation Summary

During this phase, we implemented:

1. **Explicit Causality Blocks**: `tx send`, `replay`, and `snapshot` now output visually structured causality blocks (Execution Scope, Authority, Projection Updated, Replay Status).
2. **No More Silent Failures**: `tx send` explicitly warns `[DRY RUN]` and explains that no artifacts were written unless `--yes` or valid networks are used.
3. **`hardkas explain`**: A new command providing deep, pedagogical text summaries of any artifact's deterministic integrity.
4. **Onboarding UX**: `hardkas new` now generates a `FIRST_STEPS.md` guide forcing developers to understand the conceptual model (Artifact vs Projection) within 5 minutes.
5. **Dashboard Aliasing**: `hardkas up` boots the dev server and explicitly prints the dashboard URL and runtime configuration. `hardkas ui` aliases to `hardkas dashboard`.
6. **Semantic Error Boundaries**: `REPLAY_DIVERGED` and other invariant violations now use a highly readable `Cause / Invariant Violated / Consequence / Remediation` format.

## 3. Exit Criteria

The runtime is no longer magical at the boundaries. It explicitly narrates what it does, why it does it, and where the truth lives. 

**HardKAS P1.9 is complete.**
