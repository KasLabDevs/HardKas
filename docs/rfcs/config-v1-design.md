# RFC: HardKas Config v1

## 1. Executive Summary
This RFC proposes a redesign of the HardKas configuration system for version v1. The current system, while functional for an alpha stage, has limitations in validation, mixes execution domains (L1/L2), and lacks a formal abstraction for environment profiles.

Config v1 is based on the principles of domain separation (L1 vs L2), signer abstraction (Account Abstraction), and runtime validation. This design ensures that `hardkas.config.ts` is the single source of truth for the CLI and SDK, eliminating ambiguities in multi-layer networks and reinforcing security by referencing secrets instead of inline values. Execution via TypeScript is maintained for maximum flexibility, emphasizing that **hardkas.config.ts is trusted code. Do not run HardKas inside untrusted repositories.**

## 2. Goals
- **Minimal Surface Area**: Compact and easy-to-understand initial configuration.
- **Strong TypeScript Inference**: Leverage the type system for advanced autocompletion.
- **Runtime Validation**: Structural validation via schemas (e.g., Zod) at startup.
- **Explicit L1/L2 Separation**: Differentiated domains for Kaspa (UTXO) and Igra (EVM-based).
- **Provider Abstraction**: Decouple network definition from node/RPC implementation.
- **Account Abstraction**: Flexible identity mapping (simulated, keystore, env).
- **Environment Profiles**: Environment management (dev, testnet, prod) without duplication.
- **Safe Defaults**: Secure and sensible "out of the box" values.
- **No Inline Private Keys**: Prohibit or discourage plaintext private keys.
- **Generated Docs from Schema**: Ability to generate technical documentation from the schema.
- **Future Plugin Compatibility**: Base prepared for third-party extensions.

## 3. Non-Goals
- **Full Plugin System**: The plugin system will be addressed in a subsequent RFC.
- **Enterprise Secret Manager**: HardKas does not aim to replace tools like HashiCorp Vault.
- **Executing EVM on Kaspa L1**: L1 remains a pure UTXO layer.
- **Resolving bridge trust assumptions**: Config reports status, it does not guarantee bridge security.
- **Replacing keystore implementation**: The current encrypted file system is maintained.
- **Unnecessary breaking changes**: A migration path from v0.2 will be provided.
- **Hardhat Clone**: HardKas maintains its own identity focused on Kaspa.

## 4. Design Principles

### 4.1 Minimal by default
The configuration generated by `hardkas init` should be under 20 lines, exposing only the essentials to get started.

### 4.2 Explicit over magic
Critical networks (Mainnet) must be explicitly defined. Mainnet usage should not be inferred by omission.

### 4.3 L1 and L2 are different execution domains
Kaspa L1 (UTXO/DAG) and Igra L2 (EVM/Rollup) have fundamentally different account, transaction, and receipt models. The schema must reflect this separation.

### 4.4 Secrets are references, not values
The configuration should reference secrets using environment variable names or keystore aliases. Never sensitive inline values.

### 4.5 Runtime validation complements TypeScript
While `defineHardkasConfig` provides IDE assistance, runtime validation is mandatory to ensure integrity before any network operation.

### 4.6 CLI and SDK share the same config model
A single schema governs the behavior of the command-line tool and programmatic usage via the SDK.

## 5. Proposed Top-Level Shape

```ts
export default defineHardkasConfig({
  version: 1,

  // Default active profile if --profile is not specified
  defaultProfile: "dev",

  // Environment profiles that orchestrate network and account usage
  profiles: {
    dev: {
      l1: "simnet",
      l2: "igraDev",
      accounts: ["alice", "bob"],
      defaultAccount: "alice"
    },
    testnet: {
      l1: "kaspaTestnet",
      l2: "igraTestnet",
      accounts: ["deployer"]
    }
  },

  // Kaspa L1 network definitions
  l1: {
    simnet: { kind: "simulated", network: "simnet" },
    kaspaTestnet: { 
      kind: "kaspa-rpc", 
      network: "testnet-10", 
      rpcUrl: process.env.KASPA_RPC_URL 
    }
  },

  // Igra L2 network definitions
  l2: {
    igraDev: { 
      kind: "igra", 
      l1: "simnet", 
      chainId: 19416, 
      rpcUrl: "http://localhost:8545" 
    },
    igraTestnet: { 
      kind: "igra", 
      l1: "kaspaTestnet", 
      chainId: 19417, 
      rpcUrl: "https://rpc.igra.testnet" 
    }
  },

  // Central account and identity registry
  accounts: {
    alice: { kind: "simulated" },
    deployer: { kind: "keystore", ref: "deployer" }
  },

  // Project path configuration
  paths: {
    artifacts: ".hardkas/artifacts",
    queryStore: ".hardkas/query.sqlite",
    keystore: ".hardkas/keystore"
  }
});
```

**Justification for the structure**:
- **profiles**: Allows changing the entire context (networks + accounts) with a single flag.
- **l1 / l2**: Separates infrastructure responsibilities. L2 references L1.
- **accounts**: Centralizes who signs, regardless of the network.
- **paths**: Eliminates hardcoded paths, enabling CI setups and monorepos.

## 6. L1 Network Model

```ts
type L1Network =
  | {
      kind: "simulated";
      network: "simnet";
      provider?: "in-memory" | "localnet";
    }
  | {
      kind: "kaspa-node";
      network: "simnet" | "testnet-10" | "mainnet";
      rpcUrl: string;
      dataDir?: string;
      binaryPath?: string;
    }
  | {
      kind: "kaspa-rpc";
      network: "simnet" | "testnet-10" | "mainnet";
      rpcUrl: string;
      timeoutMs?: number;
      headers?: Record<string, string>;
    };
```
- **mainnet safety**: Mainnet networks require explicit confirmation or additional security flags.
- **rpcUrl requirements**: Protocol validation (ws/wss/http/https).

## 7. L2 Network Model

```ts
type L2Network =
  | {
      kind: "igra";
      execution: "evm";
      basedOn: "kaspa";
      l1: keyof Config["l1"]; // Reference to a defined L1 network
      chainId: number;
      rpcUrl: string;
      explorerUrl?: string;
      currencySymbol?: string;
      bridge?: {
        mode: "pre-zk" | "mpc" | "zk";
        assumptions?: string[];
      };
    };
```
- **L2 references L1**: Crucial for orchestrating withdrawals and bridge validation.
- **Bridge info**: Purely informative for CLI UI and risk auditing.

## 8. Provider Abstraction

**v1 Recommendation: Option A (Embedded Provider)**.
To maintain initial simplicity, the provider definition (RPC URL, auth) lives within the network.
- **Pros**: Flatter and more readable configuration for small projects.
- **Cons**: Duplication if several networks use the same endpoint with different parameters.
- **Evolution**: In v2, a provider `registry` will be considered if complexity warrants it.

## 9. Account Abstraction

```ts
accounts: {
  alice: { kind: "simulated" },
  deployer: {
    kind: "keystore",
    ref: "deployer-key",
    networks: ["kaspaTestnet", "igraTestnet"] // Optional restriction
  },
  ci: {
    kind: "env",
    address: "kaspa:...",
    privateKeyEnv: "KASPA_PRIVATE_KEY"
  }
}
```
- **No inline keys**: The schema will prohibit fields like `privateKey: "0x..."`.
- **Redaction**: The SDK must hide environment variable values in any diagnostic log or `config show` command.

## 10. Environment Profiles
Profiles are context orchestrators:
- **Default Profile**: Determines which networks and accounts are used by default.
- **HARDKAS_PROFILE**: Environment variable to override the profile.
- **Strict Mode**: If a profile is `strict`, accounts not explicitly declared in that profile are not allowed.

## 11. Paths Model
- **CI-safe paths**: Allows pointing to temporary directories in CI.
- **gitignore**: The `init` command will generate a `.gitignore` based on these paths.

## 12. Runtime Validation
The use of **Zod** is proposed for:
- Human error messages (e.g., "networks.testnet.rpcUrl is missing").
- Error aggregation (show all config problems at once).
- **Redaction**: Ensure loaded secrets are not printed in errors.
- `hardkas config validate` and `hardkas config doctor` commands.

## 13. Typed Config Inference
Exact literal inference (e.g., so that `keyof Config["l1"]` returns the actual network names) is preserved using **const type parameters** in the `defineHardkasConfig` function. This prevents TypeScript from "widening" key types to generic `string`.

```ts
export function defineHardkasConfig<const T extends HardkasConfig>(config: T): T {
  return config;
}
```

## 14. Migration From Current Config
**Migration path**:
- **v0.2 Adapter**: The loader will detect if the `version: 1` field is missing and apply a compatibility adapter.
- **Deprecation**: A warning will be emitted in the CLI suggesting to run `hardkas config migrate`.
- **Automated Migration**: The `config migrate` command will rewrite the file (via `jiti` and AST manipulation) to the new format.

## 15. CLI Changes Required
- `--profile <name>`: Global flag.
- `hardkas config validate`: New command.
- `hardkas config doctor`: Extended to validate each profile's connectivity.
- Execution commands (`tx`, `node`, `l2`): Must be "profile-aware".

## 16. SDK Changes Required
- `Hardkas.open({ profile: "dev" })`.
- `sdk.l1` and `sdk.l2` as separate namespaces.
- Typed resolution of networks and accounts.

## 17. Security Model

> [!IMPORTANT]
> **hardkas.config.ts is trusted code. Do not run HardKas inside untrusted repositories.** HardKas executes this file during startup.

- **Mainnet Confirmation**: Profiles associated with Mainnet must activate `requireConfirmations: true` by default.
- **Secrets redaction**: Ensure the SDK does not accidentally expose environment variable values in standard output.

## 18. Example Configs

### Minimal dev config
```ts
export default defineHardkasConfig({
  version: 1,
  defaultProfile: "dev",
  profiles: {
    dev: { l1: "simnet", accounts: ["alice", "bob"] }
  },

    simnet: { kind: "simulated", network: "simnet" }

```

## 19. Open Questions
1. Should the provider registry be independent to support infrastructure plugins?
2. Should L2 security assumptions live in config or be downloaded from an official registry?
3. Support automatic interpolation of `${ENV_VAR}` in strings or leave it to TS logic?
4. Disable broadcast on Mainnet by default in all profiles unless explicitly enabled?
