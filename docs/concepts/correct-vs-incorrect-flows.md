# Correct Vs Incorrect Execution Flows

HardKas transactions move through a small number of explicit states. The local-first happy path is simple: plan, sign, then simulate/send.

## Correct Flow

```text
TxPlan -> SignedTx -> Receipt
```

CLI example:

```bash
hardkas tx plan --from alice --to bob --amount 1 --network simulated --out plan.json
hardkas artifact inspect plan.json
hardkas artifact verify plan.json --strict
hardkas tx sign plan.json --account alice --out signed.json
hardkas tx send signed.json --network simulated --yes
```

SDK example:

```typescript
const plan = await sdk.tx.plan({
  from: "alice",
  to: "bob",
  amount: "1",
  network: "simulated"
});

const signed = await sdk.tx.sign(plan, { account: "alice" });
const receipt = await sdk.tx.simulate(signed);
```

The signer does not contact the node. The sender does not touch private keys. Each step consumes the artifact from the previous step.

## Incorrect: Signing Without Inspecting

If a plan is generated by automation, inspect and verify it before signing.

```bash
hardkas artifact inspect plan.json
hardkas artifact verify plan.json --strict
```

The signer authorizes exactly what the plan says. If the destination or amount has been changed, verification should fail before a key signs.

## Incorrect: Mixing Networks

Do not plan for one network and send through another.

Example:

```bash
hardkas tx send signed.json --network mainnet --provider rpc --url ws://...
```

If `signed.json` was created for `simulated`, `simnet`, or `testnet-10`, the network context no longer matches. HardKas should abort before submitting to RPC.

## Incorrect: Using The Wrong Provider

A simulated artifact belongs to the simulated backend. A real RPC-backed artifact belongs to the selected node network. Keep provider mode explicit when crossing into real simnet:

```bash
hardkas tx send simnet-signed.json \
  --network simnet \
  --provider rpc \
  --url ws://127.0.0.1:18210 \
  --yes
```

## Incorrect: Editing Artifacts

Do not edit a `txPlan`, `signedTx`, or receipt file by hand. Changing meaningful fields such as amount, address, payload, or network invalidates the artifact.

Regenerate from the previous valid step instead.

## Incorrect: Oversized UTXO Consumption

Large mining wallets can contain many small UTXOs. Planning must select only the required inputs and fail cleanly if a single transaction would exceed mass limits. Use consolidation before attempting large sends from highly fragmented wallets.
