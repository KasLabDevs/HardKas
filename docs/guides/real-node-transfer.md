# Real rusty-kaspad Simnet Transfer

This guide walks through executing a complete transaction lifecycle against a real `rusty-kaspad` node running in `simnet` mode. This is the canonical workflow for integrating HardKAS with a live Kaspa backend.

## 1. Environment Preparation

Before beginning, ensure that your `rusty-kaspad` node is running and exposing the JSON-RPC interface.

Configure your `hardkas.config.ts`:

```typescript
import { defineConfig } from '@hardkas/cli';

export default defineConfig({
  network: 'simnet',
  provider: {
    type: 'rpc',
    url: 'http://127.0.0.1:16110' // Default rusty-kaspad simnet RPC port
  }
});
```

Ensure you have a funded account in your `.hardkas/accounts.real.json` keystore. If this account was generated by a local miner, ensure that the mining node has processed enough blocks to surpass the coinbase maturity threshold, otherwise the UTXOs will be visible but unspendable.

## 2. Planning the Transaction

The Planner contacts the node, discovers UTXOs, and generates a deterministic payload.

```bash
hardkas tx plan --from kaspa:sim_my_miner --to kaspa:sim_destination_wallet --amount 100
```

**What HardKAS does internally:**
1. Connects to `http://127.0.0.1:16110` and requests `getUtxosByAddress`.
2. Validates the network context (`simnet`).
3. Executes the largest-first UTXO selection algorithm to collect enough Kaspa.
4. Generates a `txPlan` artifact on disk (e.g., `.hardkas/artifacts/txPlan-abcdef123.json`).

## 3. Auditing the Plan (Optional but Recommended)

Before signing, inspect the deterministic artifact to ensure no man-in-the-middle has mutated the payload.

```bash
hardkas artifact inspect .hardkas/artifacts/txPlan-abcdef123.json
```

## 4. Signing

The Signer takes the plan and authorizes it using your private key.

```bash
hardkas tx sign .hardkas/artifacts/txPlan-abcdef123.json --account kaspa:sim_my_miner
```

**What HardKAS does internally:**
1. Reads the `txPlan` and verifies its SHA-256 hash.
2. Prompts for the keystore password (if encrypted) or loads the plaintext key.
3. Instantiates the WASM `PrivateKey` object in an isolated memory buffer.
4. Calculates the Kaspa Sighash and generates the cryptographic signature.
5. Destroys the WASM `PrivateKey` object from memory.
6. Writes a `signedTx` artifact to disk, which contains a cryptographic lineage pointer back to the `txPlan`.

## 5. Broadcast

Submit the signed payload to the `rusty-kaspad` node.

```bash
hardkas tx send .hardkas/artifacts/signedTx-987654fedcba.json
```

**What HardKAS does internally:**
1. Reads the `signedTx` and verifies its hash and lineage.
2. Invokes the `submitTransaction` RPC command on the real node.
3. If successful, generates a `receipt` artifact containing the final network `txId`.
4. (Optional) If polling is enabled, it monitors the node until the transaction is confirmed in a block DAA (Difficulty Adjustment Algorithm) window.

## Conclusion

You have successfully executed a real Kaspa transfer. The boundaries between network discovery (plan), cryptographic authorization (sign), and network execution (send) were strictly maintained.
