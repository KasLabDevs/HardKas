# HardKAS Determinism Policy

HardKAS is a strictly deterministic local-first transaction runtime. To guarantee that simulations, transaction plans, and artifact hashes are reproducible across any OS, machine, and timezone, the runtime enforces a strict deterministic policy.

## Prohibited Global APIs in Canonical Runtime

The following ambient state and side-effecting APIs are **STRICTLY PROHIBITED** in any package that generates or processes canonical artifacts (`core`, `artifacts`, `localnet`, `query`, `tx-builder`):

- `Date.now()` and `new Date()` (Breaks temporal determinism).
- `Math.random()` (Breaks entropy determinism).
- `crypto.randomUUID()` (Allowed ONLY for ephemeral context IDs, NOT for canonical artifact IDs).
- `String.prototype.localeCompare()` (Breaks cross-platform ordering).

## Required Alternatives

- **Sorting**: ALWAYS use `deterministicCompare` from `@hardkas/core` instead of `localeCompare()`.
- **Clock**: Retrieve time exclusively from the injected `RuntimeContext.clock.now()` or the deterministic Ledger Epoch.
- **Randomness**: Retrieve entropy exclusively from `RuntimeContext.random.next()` or by deterministically hashing inputs (e.g., `calculateContentHash(payload)`).
- **IDs**: Artifact and Workflow IDs MUST be content-addressed (derived from a hash of their contents).

## Canonical Hashing (`canonical.ts`)

The artifact hasher is strictly enforced:
- `Map`, `Set`, and `Date` instances will **THROW**. They must be explicitly serialized into plain arrays or ISO strings *before* hashing.
- Objects with non-plain prototypes (custom classes) will **THROW**.

By adhering to this policy, we ensure that a `TxPlan` or `Snapshot` generated by Developer A on Linux produces the exact same SHA-256 hash as Developer B on Windows.

## Watchers as Optimization Hints

In HardKAS, filesystem watchers (`fs.watch`, `chokidar`) are strictly **optimization hints**, never correctness primitives.
- Watcher events are treated purely as invalidation signals.
- Missed watcher events (e.g., due to ENOSPC on Linux) must always be safely recoverable via periodic generation scans or targeted syncs.
- Workspaces and the Dev Server must guarantee identical deterministic state regardless of whether a watcher fired or a fallback polling interval was used.
