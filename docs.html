What is HardKAS?

HardKAS is the ultimate offline developer suite for the Kaspa ecosystem. It gives protocol developers and smart contract engineers a secure, local environment that mimics both Kaspa L1 GHOSTDAG consensus and Igra L2 scaling execution layers.

The Three Core Pillars of HardKAS

1. Complete Offline Determinism

Mock blockDAG topologies, network latencies, and RPC behaviors locally on your computer with perfect reproducibility.

2. Decoupled Transaction Lifecycle

Create transaction plans, sign them offline in safe enclaves, broadcast separately, and audit state transitions with clear receipts.

3. Layer 2 Rollup Adapter

Build on EVM-compliant rollup structures bound securely to the robust Kaspa network using deterministic local bridges.

Safe Sandbox Principles: HardKAS is structured strictly as a local laboratory to prevent accidental mainnet private key exposures or raw coin leaks during heavy protocol testing.

HardKAS Architecture

HardKAS utilizes a decoupled architecture separating consensus, event logging, and local state query indexing systems.

Subsystem Data Pipeline

1. Core Subsystems (@hardkas/core)

Emit core events (e.g., rpc.error, workflow.submitted) to the central Event Bus.

2. Event Logging (@hardkas/query)

Subscribes to the Event Bus and writes events to append-only events.jsonl files.

3. Relational Indexing (@hardkas/query-store)

Scans artifact JSON files and event logs to build a high-performance SQLite relational query cache.

SQLite Observability advantages: Using an index database allows HardKAS to perform complex queries on GHOSTDAG block reorgs and causality lineage chains in sub-milliseconds without scanning massive file systems.

Shortcut Mode vs Pipeline Workflow

The difference is simply in the arguments you pass to the hardkas tx send command. HardKAS automatically detects what you want to do based on what you pass.

1. Shortcut Mode (1-Step)

You don't pass any file to the command. Instead, you specify who sends, who receives, and how much using flags:

hardkas tx send --from alice --to bob --amount 10

Internally, the CLI sees there is no file, grabs those three flags, and executes the entire Plan → Sign → Send flow all at once.

2. Pipeline Mode (By Steps)

First, you plan (creates a JSON artifact with the plan):

hardkas tx plan --from alice --to bob --amount 10 --out plan.json

Then, you sign it:

hardkas tx sign plan.json --account alice --out signed.json

Finally, you send it. Instead of passing flags, you pass only the path to the signed file. You don't need --from, --to, or --amount because that info is already inside the JSON:

hardkas tx send signed.json

Internally, the CLI sees you passed an argument (signed.json), reads it, verifies the signatures are correct, and broadcasts it to the network directly.

Canonical Hashing Invariants

To guarantee that artifact hashes remain identical across different development operating systems, HardKAS enforces the v3 Canonical Serialization Model.

Normalization Rules

  • Unicode Normalization (NFC): Normalizes all string characters to Canonical Composition before hashing. This ensures characters like é are treated identically regardless of whether the OS decomposed them.
  • Line-Ending Normalization (LF): Normalizes all newlines to simple UNIX \n characters, preventing hash mismatches between Windows (\r\n) and Linux/macOS.
  • Alphabetical Sorting: Recursively sorts all keys inside the JSON object alphabetically prior to execution hashing.
  • Metadata Exclusion: Automatically ignores ambient runtime variables (like createdAt, rpcUrl, and signatures) from the calculated content hash.
Cross-Platform Consistency: Following these strict invariants guarantees that the same plan will generate the exact same contentHash on Windows, macOS, and Linux CI test runners.

The SQL Query Store

The SQLite-based Query Store collects event streams and indexes local JSON files to form a relational database that powers CLI and cockpit observability queries.

Operational Features

  • Rebuildable Cache: The store.db index is treated purely as a read-model cache. You can safely delete it and rebuild it completely from source artifacts via:
    hardkas query store rebuild
  • Workspace File Locking: Uses strict store.db.lock file handlers to prevent multi-process database write contamination during rapid CLI transactions.
  • Atomic Schema Migrations: Automatically executes database updates on CLI start, preventing version mismatches.

Frequently Asked Questions (FAQ)

1. Why is my JSON receipt file failing to parse? (BOM Issue)

If you redirected command stdout using the PowerShell > operator (e.g. hardkas tx receipt --json > receipt.json), Windows automatically encodes it in UTF-16 with a Byte Order Mark (BOM). This BOM breaks standard JSON parsers!

Workaround: Copy the receipt directly from the auto-generated .hardkas/receipts/ folder instead, or enforce UTF-8 in PowerShell.

2. Why am I getting "Tracing is temporarily disabled"?

The hardkas tx trace command is temporarily suspended in v0.6.0-alpha while the query store API stabilizes. Rest assured, replay verification does NOT require traces to function.

3. What is the difference between "simulated" and "simnet"?

Simulated: A lightweight, pure memory-only state model inside JS, signing and broadcasting is not supported.
Simnet: A complete Docker container blockDAG. Use this for testing the full Plan-Sign-Send lifecycle!

4. Is the Plan → Sign → Send pipeline for debugging, or can I send directly?

Both! The hardkas tx send --from alice --to bob --amount 10 command does everything in one step. The Plan → Sign → Send pipeline is for when you want to inspect each artifact separately, audit before signing, or save a plan to sign later. It's similar to using git add + git commit + git push versus doing it all at once.

5. Where do "alice" and "bob" come from?

They are simulated accounts that HardKAS creates automatically in simulated mode. Each one is pre-funded with 1000 fake KAS. They act as convenient aliases for local development—similar to Hardhat's Account #0 and Account #1, but with readable names.

6. What if I want to send to a real Kaspa address?

You can! The aliases are just for local development. For real networks, you use real addresses:
hardkas tx send --from alice --to kaspa:qr...real...address --amount 10 --network testnet-11

7. Does it accept pubkeys and JSON keypairs?

HardKAS already accepts --private-key-stdin and --private-key-env to import real keys securely. It doesn't natively accept Solana-style JSON keypairs yet, but this feature is an easy addition planned for the future.
