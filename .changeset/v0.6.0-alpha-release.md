# HardKAS v0.7.5-alpha — Deterministic Runtime Stabilization

HardKAS has achieved an architectural milestone in transaction planning and sandbox execution reliability. This release transitions the repository into a fully stable and secure local-first development operating runtime.

### Key Stabilization Milestones:

- **Deterministic Transaction Canonicalization**: Implemented multi-tiered sorting rules (candidate UTXO pre-selection, selected inputs post-selection, and canonical recipient output sorting). The generated change output is semantically isolated and appended strictly last.
- **Replay & time-travel Hardening**: Consolidated in-memory timeline state time-travel (`reconstructStateAtDaa`), rollback math, and strict verification invariants in the isolated replay engine.
- **Workstation Security Isolation**: Secured the localhost dev-server against browser-based CSRF and DNS rebinding attacks via cryptographically random bearer session tokens, custom header checks, and host whitelists.
- **Semantic Runtime States**: Unified the state machine across all indexers, query stores, and dashboard UI layers (`EMPTY`, `ACTIVE`, `PENDING`, `VERIFIED`, `DEGRADED`, `CORRUPTED`).
- **Dashboard & Observability Stabilization**: Wired Server-Sent Events (SSE) for instantaneous local synchronization with non-blocking generation-drift banners.
- **Workflow Runtime Maturity**: Implemented safe multi-step orchestrations with agent-mode sandboxing policy proofs.
- **Documentation Canonicalization**: Reorganized all documentation into concise canonical specifications (`docs/canonical/`) and operator guides (`docs/guides/`), purging historic redundancies.
